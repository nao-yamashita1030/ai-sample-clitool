# 運用計画書

## 1. デプロイ計画

### 1.1 デプロイ環境

本アプリケーションはElectronベースのデスクトップアプリケーション（Windows対応）のため、サーバー環境へのデプロイは不要です。ユーザーのWindows環境に直接インストールして使用します。

#### 配布形式
- **インストーラー形式**: Windowsインストーラー（.exe）として配布
- **ポータブル形式**: 実行ファイル（.exe）として配布（オプション）
- **配布方法**: 
  - GitHub Releases
  - 公式Webサイトからのダウンロード
  - その他の配布チャネル（必要に応じて）

#### システム要件
- **OS**: Windows 10以降（64bit）
- **メモリ**: 4GB以上推奨
- **ディスク容量**: 100MB以上（アプリケーション本体 + データベース）
- **ネットワーク**: インターネット接続（リンク切れ調査のため）

### 1.2 デプロイ手順

#### 1.2.1 ビルド手順

1. **依存関係のインストール**
   ```bash
   npm install
   ```

2. **ビルドの実行**
   ```bash
   npm run build
   ```

3. **パッケージング**
   ```bash
   npm run package
   ```

4. **インストーラーの作成**
   ```bash
   npm run dist
   ```

#### 1.2.2 配布前チェック

- [ ] ビルドが正常に完了していること
- [ ] インストーラーが正常に作成されていること

#### 1.2.3 配布手順

2. **公式Webサイトへの公開**（該当する場合）
   - ダウンロードページにリンクを追加

3. **告知**
   - リリース通知を送信（該当する場合）

### 1.3 デプロイスケジュール

- **リリース内容**: v1.0.0（初回リリース）

- **リリース頻度**: 必要に応じて（バグ修正、機能追加時）

- **リリース期限**: 発見後1週間以内（可能な限り）

## 2. 監視・運用体制

### 2.1 監視項目

#### アプリケーション監視

本アプリケーションはデスクトップアプリケーションのため、サーバー監視は不要です。ただし、以下の項目を監視・収集します：

1. **エラーログ**
   - アプリケーションクラッシュ
   - 予期しないエラー
   - ネットワークエラー
   - データベースエラー

2. **パフォーマンス情報**
   - メモリ使用量
   - CPU使用率
   - 処理時間（リンク切れ調査の処理時間）

3. **使用状況**
   - アプリケーションの起動回数
   - 機能の使用頻度
   - 調査実行回数

4. **フィードバック**
   - ユーザーフィードバック
   - バグ報告
   - 機能要望

### 2.2 監視ツール

#### ローカルログ

- **ログファイル**: ユーザーのローカル環境にログファイルを保存
- **ログレベル**: ERROR, WARN, INFO, DEBUG
- **ログ形式**: JSON形式またはテキスト形式
- **ログ保存場所**: 
  - Windows: `%APPDATA%\アプリケーション名\logs\`
  - ログファイル名: `app-YYYY-MM-DD.log`

#### エラーレポート

- **エラーレポート機能**: クラッシュ時や重大なエラー発生時にエラーレポートを収集
- **収集内容**: 
  - エラーメッセージ
  - スタックトレース
  - システム情報（OS、メモリ、CPU）
  - アプリケーションのバージョン
- **送信方法**: 
  - ユーザーの同意を得た上で、GitHub Issuesまたは専用のエラーレポートシステムに送信
  - または、ユーザーが手動でエラーレポートを送信

#### アナリティクス（オプション）

- **使用状況の収集**: ユーザーの同意を得た上で、使用状況を収集
- **収集内容**: 
  - アプリケーションの起動回数
  - 機能の使用頻度
  - エラー発生率
- **プライバシー**: 個人を特定できる情報は収集しない

### 2.3 アラート設定

#### エラーアラート

- **重大なエラー**: クラッシュやデータ損失が発生した場合、即座に通知
- **通知方法**: 
  - GitHub Issuesへの自動投稿（該当する場合）
  - 開発チームへのメール通知（該当する場合）

#### パフォーマンスアラート

- **メモリ使用量**: メモリ使用量が異常に高い場合にアラート
- **処理時間**: 処理時間が異常に長い場合にアラート

## 3. バックアップ・リカバリ

### 3.1 バックアップ方針

本アプリケーションはデスクトップアプリケーションのため、ユーザーのローカル環境にデータが保存されます。バックアップはユーザーが手動で実施することを基本としますが、アプリケーション側でもバックアップ機能を提供します。

#### バックアップの基本方針

- **自動バックアップ**: アプリケーション側で自動バックアップ機能を提供（オプション）
- **手動バックアップ**: ユーザーが手動でバックアップを実施可能
- **バックアップ形式**: SQLiteデータベースファイルのコピー、またはエクスポート機能（CSV形式）

### 3.2 バックアップ対象

#### データベースファイル

- **ファイル名**: `app.db`（SQLiteデータベース）
- **保存場所**: 
  - Windows: `%APPDATA%\アプリケーション名\data\app.db`
- **内容**: 
  - 調査結果
  - 調査履歴
  - 設定情報

#### 設定ファイル

- **ファイル名**: `settings.json`（該当する場合）
- **保存場所**: 
  - Windows: `%APPDATA%\アプリケーション名\settings.json`
- **内容**: 
  - アプリケーション設定
  - ユーザー設定

#### ログファイル

- **ファイル名**: `app-YYYY-MM-DD.log`
- **保存場所**: 
  - Windows: `%APPDATA%\アプリケーション名\logs\`
- **内容**: 
  - アプリケーションログ
  - エラーログ

### 3.3 バックアップスケジュール

#### 自動バックアップ（オプション）

- **バックアップ頻度**: 週次（毎週日曜日）
- **バックアップタイミング**: アプリケーション起動時、または指定した時刻
- **バックアップ保持数**: 最新5件

#### 手動バックアップ

- **バックアップタイミング**: ユーザーが任意のタイミングで実施
- **バックアップ方法**: 
  - アプリケーションの「バックアップ」機能を使用
  - または、データベースファイルを手動でコピー

### 3.4 バックアップ保持期間

- **自動バックアップ**: 最新5件を保持（それ以前は自動削除）
- **手動バックアップ**: ユーザーが管理（無期限）

### 3.5 リカバリ手順

#### データベースファイルのリカバリ

1. **バックアップファイルの確認**
   - バックアップファイルの場所を確認
   - バックアップファイルが存在することを確認

2. **アプリケーションの停止**
   - アプリケーションを完全に終了

3. **データベースファイルの復元**
   - 現在のデータベースファイルを別名で保存（念のため）
   - バックアップファイルをデータベースファイルの場所にコピー

4. **アプリケーションの起動**
   - アプリケーションを起動
   - データが正常に復元されていることを確認

#### エクスポートデータからのリカバリ

1. **CSVファイルの確認**
   - エクスポートしたCSVファイルを確認

2. **インポート機能の使用**
   - アプリケーションの「インポート」機能を使用（該当する場合）
   - または、CSVファイルを手動で処理

### 3.6 リカバリテスト計画

#### リカバリテスト

- **テスト頻度**: 四半期ごと（3ヶ月に1回）
- **テスト内容**: 
  - バックアップファイルからのリカバリ
  - エクスポートデータからのリカバリ
  - データの整合性確認

#### テスト手順

1. **テスト環境の準備**
   - テスト用のデータベースを作成
   - テストデータを投入

2. **バックアップの作成**
   - 自動バックアップまたは手動バックアップを実施

3. **データの削除**
   - テストデータを削除

4. **リカバリの実施**
   - バックアップファイルからリカバリ

5. **データの確認**
   - データが正常に復元されていることを確認
   - データの整合性を確認

## 4. 障害対応手順

### 4.1 障害レベル定義

#### 障害レベルの分類

| レベル | 名称 | 説明 | 対応期限 |
|--------|------|------|----------|
| P1 | 緊急 | アプリケーションが起動しない、データ損失が発生している | 24時間以内 |
| P2 | 高 | 主要機能が使用できない、重大なバグ | 1週間以内 |
| P3 | 中 | 一部機能が使用できない、軽微なバグ | 1ヶ月以内 |
| P4 | 低 | 機能要望、UI改善 | 次回リリース時 |

#### 障害レベルの判定基準

- **P1（緊急）**: 
  - アプリケーションが起動しない
  - データベースが破損している
  - データ損失が発生している
  - セキュリティ脆弱性が発見された

- **P2（高）**: 
  - リンク切れ調査が実行できない
  - 調査結果が表示できない
  - データが保存できない
  - 重大なバグ

- **P3（中）**: 
  - 一部機能が正常に動作しない
  - UIの問題
  - 軽微なバグ

- **P4（低）**: 
  - 機能要望
  - UI改善
  - ドキュメントの改善

### 4.2 障害検知方法

#### ユーザーからの報告

- **報告チャネル**: 
  - GitHub Issues
  - メールサポート（該当する場合）
  - エラーレポート機能

#### エラーログの監視

- **ログファイルの確認**: 定期的にエラーログを確認
- **エラーレポートの確認**: エラーレポートを定期的に確認

#### 自動検知（該当する場合）

- **クラッシュレポート**: アプリケーションクラッシュ時に自動でレポートを送信
- **エラーレポート**: 重大なエラー発生時に自動でレポートを送信

### 4.3 障害対応フロー

#### 障害発生時の対応フロー

1. **障害の検知**
   - ユーザーからの報告を受領
   - エラーログを確認
   - エラーレポートを確認

2. **障害の確認**
   - 障害の再現を試みる
   - 影響範囲を特定
   - 障害レベルを判定

3. **対応の実施**
   - 緊急対応（該当する場合）
   - 修正の実施
   - テストの実施

4. **リリース**
   - 修正版のリリース
   - ユーザーへの通知

5. **事後対応**
   - 障害の原因分析
   - 再発防止策の検討
   - ドキュメントの更新

### 4.4 エスカレーション基準

#### エスカレーションの基準

- **P1（緊急）**: 即座にエスカレーション
- **P2（高）**: 1日以内にエスカレーション
- **P3（中）**: 1週間以内にエスカレーション
- **P4（低）**: 次回リリース時に検討

#### エスカレーション先

- **開発チームリーダー**: P1、P2の障害
- **開発チーム**: P3、P4の障害

### 4.5 障害対応マニュアル

#### よくある障害と対応方法

##### 障害1: アプリケーションが起動しない

- **原因**: 
  - データベースファイルの破損
  - 設定ファイルの破損
  - 依存関係の不足

- **対応方法**: 
  1. エラーログを確認
  2. データベースファイルを確認
  3. 設定ファイルを確認
  4. 必要に応じて、データベースファイルをバックアップから復元
  5. それでも解決しない場合、アプリケーションを再インストール

##### 障害2: リンク切れ調査が実行できない

- **原因**: 
  - ネットワーク接続の問題
  - タイムアウト設定の問題
  - URL形式の問題

- **対応方法**: 
  1. ネットワーク接続を確認
  2. タイムアウト設定を確認
  3. URL形式を確認
  4. エラーログを確認

##### 障害3: データベースエラー

- **原因**: 
  - データベースファイルの破損
  - ディスク容量の不足
  - ファイル権限の問題

- **対応方法**: 
  1. データベースファイルを確認
  2. ディスク容量を確認
  3. ファイル権限を確認
  4. 必要に応じて、データベースファイルをバックアップから復元

### 4.6 インシデント管理

#### インシデントの記録

- **記録方法**: GitHub Issuesを使用
- **記録内容**: 
  - インシデントの概要
  - 発生日時
  - 影響範囲
  - 対応内容
  - 解決日時
  - 再発防止策

#### インシデントの分析

- **分析頻度**: 四半期ごと（3ヶ月に1回）
- **分析内容**: 
  - インシデントの傾向
  - よくある問題
  - 改善点

## 5. 保守計画

### 5.1 保守方針

#### 保守の基本方針

- **継続的な改善**: バグ修正、機能追加、パフォーマンス改善を継続的に実施
- **セキュリティ対策**: セキュリティパッチを定期的に適用
- **ドキュメントの更新**: ドキュメントを最新の状態に保つ
- **ユーザーサポート**: ユーザーからの質問や要望に対応

### 5.2 定期メンテナンス

#### メンテナンススケジュール

- **月次メンテナンス**: 毎月1回
  - 依存関係の更新
  - セキュリティパッチの適用
  - バグ修正

- **四半期メンテナンス**: 3ヶ月に1回
  - 機能追加の検討
  - パフォーマンス改善
  - ドキュメントの更新

- **年次メンテナンス**: 1年に1回
  - 大規模な機能追加
  - アーキテクチャの見直し
  - 技術スタックの更新検討

#### メンテナンス内容

- **依存関係の更新**: npmパッケージの更新
- **バグ修正**: 報告されたバグの修正
- **パフォーマンス改善**: パフォーマンスのボトルネックを改善
- **コードのリファクタリング**: コードの品質向上
- **テストの追加**: テストカバレッジの向上

### 5.3 セキュリティパッチ適用

#### セキュリティパッチの適用方針

- **緊急パッチ**: 重大なセキュリティ脆弱性が発見された場合、即座にパッチを適用
- **定期パッチ**: 月次メンテナンス時にセキュリティパッチを適用
- **パッチの確認**: セキュリティパッチを適用する前に、影響範囲を確認

#### セキュリティパッチの適用手順

1. **脆弱性の確認**
   - セキュリティアドバイザリを確認
   - 影響範囲を特定

2. **パッチの適用**
   - セキュリティパッチを適用
   - テストを実施

3. **リリース**
   - 修正版をリリース
   - ユーザーに通知

### 5.4 バージョンアップ計画

#### バージョン管理

- **バージョン形式**: セマンティックバージョニング（Semantic Versioning）
  - 形式: `MAJOR.MINOR.PATCH`
  - 例: `1.0.0`, `1.1.0`, `1.1.1`

- **バージョンアップの基準**: 
  - **MAJOR**: 後方互換性のない変更
  - **MINOR**: 後方互換性のある機能追加
  - **PATCH**: バグ修正

#### バージョンアップスケジュール

- **PATCH**: バグ修正時（必要に応じて）
- **MINOR**: 機能追加時（四半期ごと、または必要に応じて）
- **MAJOR**: 大規模な変更時（年次、または必要に応じて）

### 5.5 保守体制

#### 保守チーム

- **開発者**: 1名以上
- **役割**: 
  - バグ修正
  - 機能追加
  - セキュリティパッチの適用
  - ドキュメントの更新

#### 保守の優先順位

1. **P1（緊急）**: 即座に対応
2. **P2（高）**: 1週間以内に対応
3. **P3（中）**: 1ヶ月以内に対応
4. **P4（低）**: 次回リリース時に検討

## 6. ログ管理

### 6.1 ログ収集方針

#### ログの基本方針

- **ログレベル**: ERROR, WARN, INFO, DEBUG
- **ログ形式**: JSON形式またはテキスト形式
- **ログの保存**: ローカルファイルに保存

#### ログの収集内容

- **ERROR**: エラー、例外、クラッシュ
- **WARN**: 警告、異常な動作
- **INFO**: 重要な処理の開始・終了、ユーザー操作
- **DEBUG**: デバッグ情報（開発時のみ）

### 6.2 ログ保持期間

- **ログファイルの保持期間**: 30日間
- **ログファイルの自動削除**: 30日を超えたログファイルは自動削除
- **エラーログの保持**: エラーログは別途保持（必要に応じて）

### 6.3 ログ分析

#### ログ分析の方法

- **手動分析**: エラー発生時にログファイルを確認
- **自動分析**: エラーレポート機能を使用（該当する場合）

#### ログ分析の内容

- **エラーの傾向**: よく発生するエラーを特定
- **パフォーマンスの問題**: 処理時間が長い処理を特定
- **ユーザーの使用状況**: 機能の使用頻度を分析

### 6.4 ログアーカイブ

#### ログアーカイブの方針

- **アーカイブ対象**: エラーログ、重要なログ
- **アーカイブ方法**: ユーザーが手動でアーカイブ（該当する場合）
- **アーカイブ形式**: ZIP形式（該当する場合）

## 7. セキュリティ運用

### 7.1 セキュリティ監視

#### セキュリティ監視の方法

- **脆弱性の監視**: 依存関係の脆弱性を定期的に確認
- **セキュリティアドバイザリの確認**: セキュリティアドバイザリを定期的に確認
- **エラーログの監視**: セキュリティ関連のエラーを監視

#### セキュリティ監視ツール

- **npm audit**: npmパッケージの脆弱性を確認
- **GitHub Dependabot**: 依存関係の脆弱性を自動で検知（該当する場合）

### 7.2 脆弱性管理

#### 脆弱性の管理方法

- **脆弱性の検知**: npm audit、GitHub Dependabotを使用
- **脆弱性の評価**: CVSSスコアを確認
- **脆弱性の対応**: セキュリティパッチを適用

#### 脆弱性対応の優先順位

- **緊急**: CVSS 9.0以上、即座に対応
- **高**: CVSS 7.0以上、1週間以内に対応
- **中**: CVSS 4.0以上、1ヶ月以内に対応
- **低**: CVSS 4.0未満、次回リリース時に検討

### 7.3 アクセス管理

#### アクセス管理の方針

本アプリケーションはデスクトップアプリケーションのため、サーバーへのアクセス管理は不要です。ただし、以下の点を考慮します：

- **データベースファイルのアクセス**: ユーザーのローカル環境に保存されるため、ユーザーが管理
- **ログファイルのアクセス**: ユーザーのローカル環境に保存されるため、ユーザーが管理

### 7.4 セキュリティインシデント対応

#### セキュリティインシデント対応フロー

1. **インシデントの検知**
   - 脆弱性の報告を受領
   - セキュリティアドバイザリを確認

2. **影響範囲の確認**
   - 脆弱性の影響範囲を特定
   - ユーザーへの影響を評価

3. **対応の実施**
   - セキュリティパッチを適用
   - テストを実施

4. **リリース**
   - 修正版をリリース
   - ユーザーに通知

5. **事後対応**
   - インシデントの原因分析
   - 再発防止策の検討

## 8. パフォーマンス管理

### 8.1 パフォーマンス監視

#### パフォーマンス監視の方法

- **ローカル監視**: ユーザーのローカル環境でパフォーマンスを監視
- **ログ分析**: ログからパフォーマンス情報を分析
- **ユーザーフィードバック**: ユーザーからのパフォーマンスに関するフィードバックを収集

#### パフォーマンス監視項目

- **メモリ使用量**: アプリケーションのメモリ使用量
- **CPU使用率**: アプリケーションのCPU使用率
- **処理時間**: リンク切れ調査の処理時間
- **UI応答性**: UIの応答時間

### 8.2 パフォーマンスチューニング

#### パフォーマンスチューニングの方針

- **ボトルネックの特定**: パフォーマンスのボトルネックを特定
- **最適化の実施**: ボトルネックを最適化
- **テストの実施**: 最適化後のパフォーマンスをテスト

#### パフォーマンスチューニングの対象

- **並列処理**: 並列リクエスト数の最適化
- **データベース**: データベースクエリの最適化
- **メモリ使用量**: メモリ使用量の削減
- **UI応答性**: UIの応答性の向上

### 8.3 キャパシティプランニング

#### キャパシティプランニングの方針

本アプリケーションはデスクトップアプリケーションのため、サーバー側のキャパシティプランニングは不要です。ただし、以下の点を考慮します：

- **データベースサイズ**: データベースファイルのサイズが大きくなりすぎないようにする
- **メモリ使用量**: メモリ使用量が大きくなりすぎないようにする
- **ディスク容量**: ディスク容量が不足しないようにする

#### キャパシティの制限

- **データベースサイズ**: 最大1GB（目安）
- **メモリ使用量**: 最大2GB（目安）
- **ログファイルサイズ**: 最大100MB/ファイル（目安）
